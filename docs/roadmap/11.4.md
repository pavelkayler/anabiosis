# 11.4. Bat scripts for manual verification

## Статус дорожной карты
- Текущий этап: `11`
- Текущий md-файл: `11.4.md`
- Статус файла: `active`
- После полного завершения файл должен быть переименован в: `11.4_done.md`

---

## Цель подэтапа
Подготовить bat-файлы для ручной верификации заказчиком.

---

## Зависимости
- `1.5.md`
- `11.1.md`
- `11.3.md`

---

## Контекст
Этот файл является частью поэтапной реализации scanner-проекта на Bybit V5.

Работа по этому файлу должна вестись только после:
1. прочтения `0.md`;
2. прочтения последнего отчёта по предыдущему roadmap-файлу;
3. закрытия всех хвостов из предыдущего отчёта.

Если хвосты предыдущего шага не закрыты, реализацию этого файла начинать нельзя.

---

## Что должно быть получено по итогам подэтапа
- backend/frontend build bats
- backend/frontend test bats
- smoke and run-all helpers

---

## Подробные задачи реализации
- Подготовить технический дизайн подэтапа и проверить, что он не конфликтует с `0.md` и уже завершёнными файлами.
- Реализовать основную цель подэтапа: Подготовить bat-файлы для ручной верификации заказчиком.
- Доставить результат: backend/frontend build bats.
- Доставить результат: backend/frontend test bats.
- Доставить результат: smoke and run-all helpers.
- Сохранить модульность и не допустить разрастания логики в один файл.
- Подготовить или обновить тесты под новый функционал.
- Подготовить отчёт и список хвостов по результату.

---

## Технические требования
- Код должен оставаться модульным и читабельным.
- Не допускается смешивать в одном большом файле unrelated logic из разных слоёв.
- Все timestamps хранить и передавать в UTC.
- Все решения, влияющие на scanner state и decision logic, должны быть объяснимыми и тестируемыми.
- Везде, где это применимо, нужно учитывать, что архитектура уже сейчас проектируется под BTCUSDT, ETHUSDT и SOLUSDT, хотя активным остаётся только BTCUSDT.

## Требования к проверкам
- Тесты писать обязательно для нового функционала, даже если агент не может запустить их сам.
- Все проверки должны быть доступны через отдельные bat-файлы.
- Отчёты должны явно отражать, что было реально прогнано, а что нет.

---

## Ограничения и дисциплина подэтапа
- Реализовать только scope этого файла.
- Не перескакивать на будущие roadmap-файлы.
- Не менять стек, архитектурные правила и workflow без отдельного согласования.
- Не использовать JSONL как основной источник данных проекта.
- Не подменять explainable statuses невнятными общими формулировками.

---

## Что считается завершением подэтапа
- Цель подэтапа достигнута: Подготовить bat-файлы для ручной верификации заказчиком.
- Deliverables реализованы и интегрированы в проектный каркас.
- Новый код не нарушает agreed modular layout и не создаёт файлов-монстров.
- Подэтап можно закрыть и переименовать в `_done` после отчёта и ручной/автоматической проверки.

---

## Проверки и верификация
- Подготовить bat-команды для build/test/smoke либо запустить проверки напрямую, если среда позволяет.
- Проверить целевой сценарий подэтапа вручную или через тесты.
- Проверить, что изменения не ломают предыдущие завершённые этапы.
- Зафиксировать результат проверки в отчёте.

---

## Что явно вне scope этого файла
- Не выполнять задачи из будущих roadmap-файлов заранее.
- Не делать лишних рефакторов вне области текущего подэтапа.
- Не создавать файлы-монстры и не нарушать agreed modular layout.

---

## Требования к отчёту после выполнения
Отчёт по этому файлу должен быть создан в `docs/reports/` и содержать:

- Что именно реализовано по файлу `11.4.md` и насколько это покрывает цель подэтапа.
- Какие файлы изменены и какие новые файлы добавлены.
- Какие команды запускались и какие результаты были получены.
- Какие хвосты остались, что не удалось проверить и что должен закрыть следующий исполнитель.
- Какой следующий логический шаг после этого подэтапа.

---

## Правило завершения файла
После полного завершения работ по этому файлу:
1. выполнить или подготовить проверки;
2. оформить отчёт;
3. перечислить хвосты, если они остались;
4. только после этого переименовать файл в `11.4_done.md`.

Если хвосты блокируют переход дальше, файл не считается закрытым.


---

## Уточнения редакции v3

### Минимальный набор bat-скриптов
Обязательны:
- docker up/down/logs;
- backend build;
- frontend build;
- backend tests;
- frontend tests;
- smoke run;
- optional replay validation run.

### Правило
Если агент не может прогнать команды сам, он обязан подготовить bat-скрипты так, чтобы заказчик смог выполнить их без ручной сборки команд из чата.
