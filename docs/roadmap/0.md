# 1. Назначение этого файла

Этот файл — главный управляющий документ для исполнителя проекта.

Его задача:

1. дать общий контекст проекта;
2. зафиксировать утверждённый стек и архитектурные решения;
3. определить строгие правила движения по roadmap-файлам;
4. определить формат общения с заказчиком;
5. определить формат отчёта после каждого подэтапа;
6. запретить хаотичную разработку, перескоки и лишние рефакторы;
7. зафиксировать единый стандарт качества реализации.

Исполнитель обязан руководствоваться этим файлом на протяжении всей работы по проекту.

# 2. Общий контекст проекта

Нужно реализовать сервис аналитического сканера для рынка крипто-деривативов на **Bybit V5**, который:

1. собирает рыночные данные;
2. сохраняет их в БД;
3. рассчитывает аналитические производные;
4. определяет рыночный режим;
5. определяет directional bias;
6. определяет тип сетапа;
7. формирует итоговое торговое решение;
8. отдаёт результат на frontend;
9. делает это в понятной, модульной, проверяемой и расширяемой архитектуре.

## Начальный рабочий scope
На старте активен только один символ:

- `BTCUSDT`

После полной стабилизации BTC-версии можно расширяться на:

- `ETHUSDT`
- `SOLUSDT`

Но архитектура проекта, БД и контракты должны **сразу проектироваться под 3 символа**, даже если реально активен пока только BTC.

# 3. Главный принцип реализации

Сначала делается **надёжное аналитическое ядро**, а не “всё сразу”.

На ранних этапах проект **не является торговым ботом**.  
Первичная цель — корректный и объяснимый **scanner + decision engine**.

То есть:

- сначала данные;
- потом аналитика;
- потом decision logic;
- потом вывод в UI;
- и только позже можно думать о дальнейших торговых слоях.

# 6. Правила хранения данных

## 6.1. БД используется с самого начала
Нельзя писать основную рабочую историю в `jsonl` как основной формат хранения проекта.

Разрешено:

- временное логирование;
- debug-логи;
- служебные event-логи;
- кратковременные технические дампы.

Но основная рыночная история, derived-метрики, сигналы и статусы должны храниться **в таблицах БД**.

---

## 6.2. Полный raw orderbook хранить не надо
Полная бесконечная запись сырого orderbook запрещена как основной подход.

Вместо этого должны сохраняться **derived book metrics**, например:

- best bid / ask
- spread
- near bid depth
- near ask depth
- imbalance
- passive walls
- compact book snapshots по необходимости

Near-depth окна, которые должны быть заложены:

- `0.10%`
- `0.25%`
- `0.50%`

---

## 6.3. Retention policy
На проекте сразу действует retention policy.

Подтверждено:

- raw trades: `24h`
- raw liquidations: `7d`
- raw ticker snapshots: `24h`
- raw minute OI snapshots: `30d`
- raw funding snapshots/history cache: `90d`
- raw orderbook derived metrics: `7d`
- 1m analytics: `90d`
- 5m / 15m analytics: `180d`
- signal decisions / reasons: без автоудаления на старте
- service events: `30d`

Детальные таблицы и политики будут описываться в отдельных roadmap-файлах.

# 8. Архитектурный стиль проекта

## 8.1. Главный принцип — модульность
Код должен быть разбит на маленькие и средние читаемые компоненты/модули.

Цель:

- удобное чтение;
- удобный поиск ошибок;
- удобное внесение патчей;
- минимизация файлов-монстров.

---

## 8.2. Ограничение по размеру файлов
Ориентир:

- нормально: до `500–700` строк на файл
- немного больше — допустимо
- огромные файлы на тысячи / десятки тысяч строк — запрещённый стиль

Исполнитель не должен складывать большую бизнес-логику в один огромный файл.

---

## 8.3. Структура frontend
Frontend должен строиться по FSD-подобной логике.

Ожидаемая идея структуры:

- `app/`
- `pages/`
- `features/`
- `shared/`

Дальнейшая детализация в рамках этапов приветствуется, если она сохраняет читаемость.

---

## 8.4. Структура backend
Backend тоже должен быть модульным и аналогично читабельным.

Ожидаемая идея структуры:

- `app/`
- `features/`
- `shared/`
- `infra/`
- `db/`
- `ws/`
- `analytics/`
- `decision/`
- дополнительные bounded folders по необходимости

Главное требование: не допускать backend-файлов-монстров и хаотичной свалки логики.

---

# 9. Принцип движения по roadmap-файлам

## 9.1. Формат этапов
Работа ведётся строго по файлам roadmap.

Главный файл:
- `0.md`

Далее:
- `1.1.md`
- `1.2.md`
- `1.3.md`
- ...
- при необходимости:
  - `6.4.1.md`
  - `6.4.2.md`
  - и так далее

---

## 9.2. Перескакивать нельзя
Исполнитель не имеет права:

- перескакивать через этапы;
- хвататься за будущие блоки заранее;
- рефакторить будущие слои “заодно”;
- менять архитектурные решения без отдельного указания.

Работа должна идти **строго по порядку**.

---

## 9.3. Правило `_done`
После того как конкретный md-файл полностью реализован, проверен, и по нему оформлен отчёт, этот файл должен быть переименован:

- `1.1.md` → `1.1_done.md`
- `2.4.md` → `2.4_done.md`

Это правило обязательно.

---

## 9.4. Правило закрытия хвостов
После завершения каждого md-файла исполнитель оставляет отчёт.

Следующий исполнитель обязан:

1. сначала прочитать предыдущий отчёт;
2. закрыть все хвосты и недоделки из него;
3. только потом переходить к следующему md-файлу.

Игнорировать хвосты из отчёта запрещено.

---

# 10. Правила общения с заказчиком

## 10.1. Как заказчик взаимодействует с исполнителем
Заказчик может:

- дать задачу;
- сказать идти по следующему md-файлу;
- прислать замечания;
- прислать результаты билдов, тестов и ручных проверок;
- вернуть реализацию на доработку.

---

## 10.2. Стиль общения исполнителя
Исполнитель обязан отвечать:

- чётко;
- по делу;
- без лишней воды;
- без абстрактных рассуждений;
- без самовольного расширения scope.

Нужно исходить из того, что проект ведётся поэтапно и управляется вручную.

---

## 10.3. Что запрещено исполнителю в общении
Запрещено:

- переписывать проект целиком, если задан точечный подэтап;
- самовольно “улучшать всё вокруг”;
- вносить лишние рефакторы;
- смешивать несколько roadmap-файлов в одну реализацию;
- скрывать недоделки;
- писать “всё готово”, если есть незакрытые хвосты.

# 12. Правила build / test / smoke

## 12.1. Build и test нужны всегда
После каждого подэтапа исполнитель должен:

- подготовить код к проверке;
- написать тесты для нового функционала;
- по возможности прогнать проверки сам;
- если не может прогнать сам — дать готовые bat-скрипты и инструкции для ручного прогона заказчиком.

---

## 12.2. Отдельные bat-файлы обязательны
В проекте должны быть отдельные bat-файлы для:

- запуска всего dev-окружения
- остановки
- поднятия docker
- остановки docker
- логов docker
- backend build
- frontend build
- backend tests
- frontend tests
- smoke-check сценариев

---

## 12.3. Если исполнитель не может выполнить проверку сам
Это не освобождает его от обязанностей.

Он всё равно должен:

1. написать тесты;
2. подготовить bat-команды;
3. описать, что именно должен запустить заказчик;
4. описать ожидаемый результат;
5. дождаться обратной связи по ручному прогону.

# 13. Формат отчёта после каждого md-файла

После каждого md-файла должен быть создан отчёт.

Рекомендуемое расположение:
- `docs/reports/`

Формат:
- `1.1_report.md`
- `2.4_report.md`
- и т.д.

## 13.1. Обязательная структура отчёта
Каждый отчёт обязан содержать:

1. что сделано;
2. какие файлы изменены;
3. какие новые файлы добавлены;
4. какие команды запускались;
5. что удалось проверить;
6. что не удалось проверить;
7. какие хвосты остались;
8. какие есть риски;
9. какой следующий логический шаг;
10. что должен проверить следующий исполнитель перед продолжением.

---

## 13.2. Честность отчёта
Отчёт не должен притворяться полным, если подэтап выполнен частично.

Если есть незакрытые хвосты, это должно быть явно перечислено.

---

# 14. Правила по UI и объяснимости статусов

## 14.1. В шапке интерфейса обязательно должно быть
- текущий символ
- статус сервиса
- статус прогрева
- текущий этап
- текущий md-файл

Это должно обновляться после каждой завершённой правки.

---

## 14.2. Статусы сигнала
На проекте используются статусы:

- `ENTER_LONG`
- `ENTER_SHORT`
- `WAIT`
- `INVALID`

---

## 14.3. WAIT и INVALID должны быть объяснимыми
Эти статусы нельзя отдавать как “немые коды”.

Для них обязательно нужны:

- machine-readable reason code
- human-readable title
- human-readable text explanation
- blockers list
- описание, чего именно система ждёт

То есть frontend должен уметь показать не только `WAIT`, но и:
- **почему WAIT**
- **чего именно ждём**
- **что мешает входу**
- **что должно случиться для перехода к valid setup**

То же касается `INVALID`:
- по какой причине данные/состояние сейчас невалидны
- что нужно для восстановления корректного анализа

---

# 15. Логика принятия решений

## 15.1. Основной принцип
Decision engine должен быть:

- **rule-based**
- детерминированным
- объяснимым

Confidence score допускается только как **вторичный атрибут**.

---

## 15.2. Confidence score
Score не должен сам по себе запускать сигнал.  
Он нужен как дополнительная интерпретация качества already-derived signal context.

---

## 15.3. На раннем этапе нужно считать уже не только направление
На первом полноценном аналитическом этапе нужно уметь рассчитывать:

- entry zone
- invalidation level
- target zone

То есть решение не должно быть слишком абстрактным.

---

# 16. Прогрев системы

Так как minute OI строится локально после запуска сервиса, на проекте обязателен warmup.

Утверждено:

- минимальный warmup runtime: `30 минут`

До окончания прогрева сервис не должен преждевременно изображать полную готовность к анализу.

---

# 17. Принцип работы с Bybit

## 17.1. Основной транспорт
Приоритет:
- **WebSocket**

REST использовать только:
- для bootstrap;
- для historical bootstrap;
- для snapshots;
- для восстановления пропусков;
- для тех участков, которые недоступны через WS.

---

## 17.2. Исторический OI
У Bybit нет официального исторического `1m OI` через REST.  
Поэтому minute OI для проекта должен локально накапливаться из live ticker state после запуска сервиса.

Это надо помнить на всех этапах реализации.

---

## 17.3. Spot и linear нельзя смешивать
Потоки данных `spot` и `linear` должны обрабатываться отдельно.

Особенно это критично для:
- trades
- CVD
- orderbook metrics
- freshness
- derived analytics

---

# 18. Общие инженерные требования

## 18.1. Время
Все timestamps:
- только в **UTC**

Преобразование в локальное время — только на UI, если понадобится.

---

## 18.2. Чистота контракта
Backend и frontend должны обмениваться понятными DTO/контрактами.  
Нельзя тащить на фронт хаотичные raw-объекты без нормализации.

---

## 18.3. Минимум магии
Нельзя строить критические решения на скрытых неочевидных эвристиках без явного описания их логики.

Если появляется rule — он должен быть:
- описан;
- реализован;
- протестирован;
- отражён в reason object.

---

## 18.4. Никакой лишней самодеятельности
Если в roadmap-файле сказано сделать определённый подэтап, исполнитель делает именно его.

Запрещено:
- тащить будущий функционал без запроса;
- менять согласованный стек;
- самовольно заменять архитектуру;
- делать неутверждённые библиотеки “потому что так удобнее”.

---

# 19. Что считается успешной работой исполнителя

Исполнитель работает правильно, если:

1. идёт строго по текущему roadmap-файлу;
2. не перескакивает вперёд;
3. реализует только согласованный scope;
4. сохраняет модульность;
5. не плодит огромные файлы;
6. пишет тесты под новый код;
7. даёт честный отчёт;
8. закрывает хвосты;
9. переименовывает завершённый roadmap-файл в `_done`;
10. поддерживает проект в состоянии, пригодном для ручных проверок заказчиком.

---

# 20. Что считается неправильной работой исполнителя

Исполнитель работает неправильно, если:

- делает огромные файлы-монстры;
- сливает несколько этапов в один;
- игнорирует правила отчёта;
- не закрывает хвосты;
- выдает partial work как finished;
- пишет патчи вместо полных файлов там, где нужен архив;
- перескакивает на будущие этапы;
- внедряет лишний стек или стороннюю архитектуру;
- не даёт объяснимых причин для WAIT/INVALID;
- ломает agreed modular layout.

---

# 21. Ожидаемая последовательность работы

Общий принцип:

1. согласовывается текущий md-файл;
2. реализуется только он;
3. готовится отчёт;
4. при необходимости заказчик гоняет build/test/smoke;
5. по итогам замечаний закрываются хвосты;
6. файл становится `_done`;
7. только после этого можно идти дальше.

---

# 22. Специальная инструкция исполнителю

Перед началом реализации каждого нового md-файла исполнитель обязан:

1. прочитать этот `0.md`;
2. проверить предыдущий отчёт;
3. выписать все хвосты;
4. закрыть хвосты;
5. только после этого переходить к реализации нового подэтапа.

Если хвосты не закрыты — следующий подэтап начинать нельзя.

# 23. Итоговая суть этого файла

`0.md` задаёт не только общий контекст проекта, но и рабочую дисциплину.

Этот проект должен развиваться как:

# 25. Дополнительные фиксированные решения по логике сканера

## 25.1. Двухслойный режим расчёта
Сервис должен поддерживать два уровня расчёта одновременно:

1. **tick-layer** — обновляется на каждом тике и пересчитывает live market state, regime preview, bias preview, setup preview, entry/invalidation/targets preview, blockers и explainability state;
2. **tradeable-decision layer** — финально фиксирует `ENTER_LONG` / `ENTER_SHORT` только на закрытии `1m` свечи.

Следствие:
- intrabar разрешены только preview-статусы;
- tradeable decision не должен считаться валидным внутри ещё не закрытой `1m` свечи в BTC v1.

## 25.2. Внешние и внутренние статусы
Внешние trade-status значения:
- `ENTER_LONG`
- `ENTER_SHORT`
- `WAIT`
- `INVALID`

Внутренние служебные статусы, которые должны существовать в backend state и UI debug-панели:
- `WARMING_UP`
- `WATCHLIST`
- `PREVIEW`
- `DEGRADED`
- `BLOCKED_BY_DATA`
- `BLOCKED_BY_GEOMETRY`

Они не заменяют внешние финальные статусы, а помогают объяснить путь принятия решения.

## 25.3. Приоритет setup над bias
Если bias сменился, а setup ещё невалиден, итоговый внешний статус должен оставаться `WAIT`.

Правило:
- regime и bias могут пересчитываться сразу;
- setup обязан быть завершённым и tradeable;
- bias без валидного setup не даёт права на `ENTER_*`.

## 25.4. Конфликт structure и flow
При конфликте слоёв используются такие приоритеты:

- **structure / levels** первичны для входной геометрии, invalidation и target logic;
- **flow metrics** первичны для подтверждения, штрафов и veto.

Нельзя спасать сломанный setup хорошим flow. Нельзя считать красивый breakout tradeable без достаточного flow confirmation.

## 25.5. Нормализация метрик
Пороговая логика не должна использовать только абсолютные значения.

Для BTC v1 каждая ключевая метрика должна, где применимо, иметь одновременно:
- raw value;
- volatility-normalized value;
- rolling z-score или percentile rank по локальной истории.

Рекомендованные rolling windows:
- fast: `15m`
- medium: `1h`
- context: `24h`

Это особенно обязательно для:
- OI delta / OI acceleration;
- futures CVD / spot CVD;
- liquidation bursts;
- price expansion;
- book imbalance.

## 25.6. OI использовать в двух формах
В аналитике одновременно должны использоваться:
- `openInterest`
- `openInterestValue`

Роли:
- `openInterestValue` — floors, symbol quality, participation context;
- `openInterest` — minute delta, acceleration, micro-dynamics.

## 25.7. CVD считать в двух версиях
Для futures и spot должен считаться:
- raw CVD по полному taker flow;
- filtered CVD с quality-filter по very-low notional noise.

Расхождение между raw и filtered версиями не является отдельным сигналом, но создаёт penalty / quality warning.

## 25.8. Формализация squeeze-prone regime
`squeeze_prone_up` и `squeeze_prone_down` нельзя определять по одному признаку.

Для `squeeze_prone_up` одновременно требуются:
1. price expansion up;
2. OI expansion up;
3. futures flow dominance up;
4. хотя бы одно дополнительное подтверждение:
   - funding extreme / funding tilt;
   - short liquidation burst;
   - supportive book imbalance;
   - явное отставание spot при агрессивном perp push.

Для `squeeze_prone_down` — зеркальная логика.

## 25.9. Setup families для BTC v1
На этапе BTC v1 должны существовать отдельные семейства setup-ов:
- `breakout_long`
- `breakout_short`
- `retest_long`
- `retest_short`
- `trap_reversal_long`
- `trap_reversal_short`
- `failed_continuation_long`
- `failed_continuation_short`
- `absorption_reversal_long`
- `absorption_reversal_short`
- `none`

Правила:
- `trap_reversal` требует именно structural fakeout;
- `failed_continuation` не должен смешиваться с trap;
- `absorption_reversal` не должен скрываться внутри trap как неявный частный случай.

## 25.10. Retest и breakout
Tradeable `retest` разрешён только после подтверждённого breakout на закрытой `1m` свече.

Ранний intrabar retest может существовать только как preview/watchlist, но не как финальный `ENTER_*`.

## 25.11. Geometry rules
Entry / invalidation / targets должны строиться от:
- structure levels;
- volatility buffer;
- mark-aware risk checks.

Роли цены:
- `markPrice` — primary price для risk logic, trigger proximity и контроля invalidation;
- `lastPrice` — secondary UI-visible companion.

Если валидные уровни entry/invalidation/targets не могут быть построены, система обязана вернуть `WAIT`, а не абстрактный `ENTER_*`.

## 25.12. Staleness budgets
Для BTC v1 нужно жёстко использовать soft/hard stale thresholds.

Стартовые бюджеты:
- ticker: soft `2s`, hard `5s`
- linear trades: soft `2s`, hard `5s`
- spot trades: soft `3s`, hard `8s`
- orderbook: soft `2s`, hard `5s`
- liquidation: soft `15s`, hard `60s`
- funding current: soft `60s`, hard `10m`

Особое правило:
- stale liquidation feed не обязан автоматически переводить систему в `INVALID`;
- он может давать `DEGRADED`, confidence penalty или `WAIT`, если конкретный setup требует liquidation confirmation.

## 25.13. Warmup gate
После рестарта tradeable decision имеет право стать валидным только после:
- успешного bootstrap;
- здоровых WS feeds;
- минимум `30m` локального minute OI;
- минимум `15m` live futures CVD;
- минимум `15m` live spot CVD;
- минимум `5m` стабильного orderbook-derived state.

До этого внешние tradeable ENTER-решения запрещены.

## 25.14. Cooldown и expiry
В BTC v1 должны существовать отдельные конфигурируемые параметры:
- `minBarsBetweenTradeableSignals`
- `cooldownAfterInvalidation`
- `cooldownAfterTarget1`
- `setupExpiryCandles`

Стартовые значения:
- `minBarsBetweenTradeableSignals = 3`
- `cooldownAfterInvalidation = 2`
- `cooldownAfterTarget1 = 2`
- `setupExpiryCandles = 15`

## 25.15. Валидация качества
Primary validation metric для tradeable сигнала:
- `target1 hit before invalidation within expiry window`

Secondary metrics:
- `MFE`
- `MAE`
- `time_to_target1`
- `time_to_invalidation`
- `target2_hit_rate`
- `false_breakout_rate`
- `trap_quality_rate`

Отдельно должна существовать оценка полезности `WAIT` / `INVALID` как фильтра.

## 25.16. Версионирование правил
Нужно хранить и показывать версии логики:
- `feature_schema_version`
- `regime_rules_version`
- `setup_rules_version`
- `trade_geometry_version`
- `explanation_rules_version`
- `threshold_profile_version`

## 25.17. Replay и validation storage
С самого начала архитектура должна предусматривать:
- signal snapshot archive;
- replay-ready storage для сравнения сигналов на истории;
- validation cases;
- reviewed outcomes;
- outcome labels для сравнения версий логики.

## 25.18. Профили по символам
BTC v1 живёт на собственном profile preset.

Позже ETHUSDT и SOLUSDT должны иметь отдельные preset-профили. Нельзя автоматически переносить BTC thresholds на ETH/SOL без отдельной калибровки.

## 25.19. UI-объяснимость
`WAIT`, `INVALID`, `WATCHLIST`, `PREVIEW`, `DEGRADED` и `BLOCKED_*` должны иметь:
- machine-readable code;
- human-readable title;
- short explanation;
- detailed explanation;
- next expected condition / what scanner waits for.

## Уточнения редакции v3: операционные правила и недостающая конкретика

### 1. Источники данных и рабочий рынок v1
Во всей первой версии проекта поддерживается только один активный торговый инструмент:
- `BTCUSDT` linear perpetual на Bybit V5.

Подтверждающий спот-инструмент:
- `BTCUSDT` spot.

Во v1 **не используются**:
- inverse contracts;
- dated futures;
- options;
- внешние биржи;
- оракулы;
- news/API внешнего контекста.

### 2. Основной принцип расчёта решения
Система работает в двух слоях:

#### Tick-layer
На каждом тике пересчитываются:
- market state;
- regime preview;
- directional bias preview;
- setup preview;
- blockers;
- WAIT/INVALID explanations.

#### Tradeable-decision layer
Финальный tradeable-статус:
- `ENTER_LONG`
- `ENTER_SHORT`

может быть выдан **только после закрытия 1m свечи**.

Во v1 для BTC запрещено выдавать tradeable `ENTER_*` intrabar.
Intrabar допускаются только служебные статусы:
- `PREVIEW`
- `WATCHLIST`
- `WAIT`
- `INVALID`

### 3. Обязательные служебные статусы
Помимо внешних финальных статусов, внутри проекта должны существовать:
- `WARMING_UP`
- `WATCHLIST`
- `PREVIEW`
- `DEGRADED`
- `BLOCKED_BY_DATA`
- `BLOCKED_BY_GEOMETRY`

Они используются:
- для отладки;
- для UI;
- для честного explanation layer;
- для диагностики состояния decision pipeline.

### 4. Обязательная формализация setup families
Во v1 набор setup families должен быть таким:
- `breakout_long`
- `breakout_short`
- `retest_long`
- `retest_short`
- `trap_reversal_long`
- `trap_reversal_short`
- `failed_continuation_long`
- `failed_continuation_short`
- `absorption_reversal_long`
- `absorption_reversal_short`
- `none`

Правила:
- `trap_reversal` требует **обязательного structural fakeout**;
- `failed_continuation` не смешивается с trap;
- `absorption_reversal` не смешивается ни с trap, ни с failed continuation;
- `retest` считается tradeable только после confirmed breakout на закрытой 1m свече.

### 5. Обязательные уровни сделки
Tradeable decision запрещён без расчёта:
- `entry zone`
- `invalidation`
- `target1`
- `target2/target zone`

Если валидную trade geometry посчитать нельзя, итог обязан быть:
- `WAIT`
или
- `BLOCKED_BY_GEOMETRY` → UI-friendly WAIT explanation.

### 6. Исторический bootstrap v1
На этапе bootstrap должны быть загружены:
- candles:
  - `1m`: `24h`
  - `5m`: `7d`
  - `15m`: `30d`
  - `1h`: `180d`
  - `4h`: `365d`
- open interest history:
  - `5m`: `30d`
  - `15m`: `90d`
  - `1h`: `180d`
  - `4h`: `365d`
- funding history:
  - `30d`
- recent trades:
  - короткое окно только для continuity check и sanity bootstrap.

Важно:
- recent trades bootstrap не заменяет live-CVD;
- tradeable решения до прогрева всё равно запрещены.

### 7. Warmup и право на tradeable signals
До получения статуса `ANALYSIS_READY` должны быть одновременно выполнены условия:
- REST bootstrap completed;
- все ключевые WS feeds healthy;
- есть минимум `30m` локального minute OI;
- есть минимум `15m` live futures CVD;
- есть минимум `15m` live spot CVD;
- есть минимум `5m` стабильного orderbook-derived state.

До этого tradeable `ENTER_*` запрещён.

### 8. Формулы метрик: обязательный минимум
#### CVD
Primary CVD считается по quote-notional:
- `buyQuote = Σ(size * price)` по taker buy;
- `sellQuote = Σ(size * price)` по taker sell;
- `deltaQuote = buyQuote - sellQuote`;
- `cvdQuote = cumulative(deltaQuote)`.

Нужно поддерживать:
- `raw_cvd`;
- `filtered_cvd` (без микро-сделок ниже low-notional floor).

#### OI
Должны использоваться обе версии:
- `openInterest`;
- `openInterestValue`.

Acceleration — это не абстрактный флаг, а разница скоростей:
- `oiDelta1m = oi_t - oi_t-1`;
- `oiAccel1m = oiDelta1m - prevOiDelta1m`;
- аналогично на `5m`.

#### Liquidation
Для каждого liquidation bucket должны рассчитываться:
- `shortLiqUsd`
- `longLiqUsd`
- `totalLiqUsd`
- `liqImbalance = shortLiqUsd - longLiqUsd`
- `liqBurstZ`

#### Book imbalance
Для каждого окна глубины:
- `bidUsd = Σ(bidSize * bidPrice)`
- `askUsd = Σ(askSize * askPrice)`
- `imbalance = (bidUsd - askUsd) / (bidUsd + askUsd)`

Окна для near-depth обязательны:
- `0.10%`
- `0.25%`
- `0.50%`

### 9. Нормализация и threshold profiles
Каждая важная метрика должна существовать в трёх формах:
- raw;
- volatility-normalized;
- local z-score / percentile rank.

Абсолютные пороги допустимы только для:
- quality floors;
- spread / liquidity limits;
- minimum participation;
- minimum RR;
- data eligibility.

Все decision thresholds должны:
- храниться в versioned config profiles;
- быть явно привязаны к symbol profile;
- логироваться в decision context;
- быть сравнимыми между версиями rules.

### 10. Staleness budgets
Обязательные soft/hard stale thresholds:

- ticker: `2s / 5s`
- linear trades: `2s / 5s`
- spot trades: `3s / 8s`
- orderbook: `2s / 5s`
- liquidations: `15s / 60s`
- funding: `60s / 10m`

Правило:
- stale liquidation feed сам по себе не делает состояние `INVALID`;
- он даёт degraded confidence или `WAIT`, если конкретный setup требует liquidation confirmation.

### 11. Dedupe и carry-forward
Обязательны:
- дедупликация trade events;
- дедупликация orderbook payloads по channel/ts/seq identity;
- carry-forward minute OI snapshot только с `stale` mark, если за минуту не пришло новое значение.

Trade и orderbook запрещено «интерполировать» искусственно.

### 12. Validation, replay и no-ML правило
Во v1:
- нет машинного обучения;
- нет retraining;
- нет model update loop.

Но обязательно должны существовать:
- replay storage;
- validation cases;
- post-factum outcome tracking;
- rule versioning;
- threshold profile versioning.

Главная метрика качества для `ENTER_*`:
- `target1 hit before invalidation within expiry window`.

### 13. Expiry и cooldown
Во v1 для BTC нужно заложить:
- `setupExpiryCandles = 15`
- `secondaryReviewWindow = 30`
- `minBarsBetweenTradeableSignals = 3`
- `cooldownAfterInvalidation = 2`
- `cooldownAfterTarget1 = 2`

Это стартовые инженерные baseline values. Они могут быть изменены позже только через versioned profile.

### 14. UX и доступ
Во v1 интерфейс:
- read-only;
- без редактирования thresholds из UI;
- без роли трейдера;
- без приватных API-ключей биржи;
- без мультиязычности.

Главное требование к UI:
- прозрачно показывать причину `WAIT`;
- прозрачно показывать причину `INVALID`;
- показывать текущий режим прогрева и деградации;
- показывать активный phase / roadmap file.

Если такие темы появятся в обсуждении, их нужно относить в future backlog, а не в обязательный scope BTC v1.

# 26. Последние обязательные уточнения редакции

В этой финальной редакции дополнительно считаются обязательными:

1. **Reconnect/backoff policy** должна быть задана явно, а не “по умолчанию библиотеки”.

4. **Benchmark validation baseline** должен существовать отдельно от основной signal-логики, чтобы оценивать добавочную ценность scanner-а.
5. **Long/short ratio** допускается только как soft feature и не должен становиться hard gate для решения.
