## Технические требования
- Код должен оставаться модульным и читабельным.
- Не допускается смешивать в одном большом файле unrelated logic из разных слоёв.
- Все timestamps хранить и передавать в UTC.
- Все решения, влияющие на scanner state и decision logic, должны быть объяснимыми и тестируемыми.
- Везде, где это применимо, нужно учитывать, что архитектура уже сейчас проектируется под BTCUSDT, ETHUSDT и SOLUSDT, хотя активным остаётся только BTCUSDT.

## Требования к рыночным данным

- Spot и linear обрабатывать строго раздельно.
- Для minute OI использовать локальное накопление после запуска сервиса.
- Любая derived-логика должна быть объяснима, тестируема и пригодна для reason object.

## Требования к explainability и UI
- `WAIT` и `INVALID` должны сопровождаться machine-readable reason codes и human-readable текстом.
- UI должен уметь показать текущий этап и текущий md-файл.
- Backend contracts и frontend rendering не должны расходиться по смыслу полей.

## Формальная логика regime classifier

### `trend_up`
Требуется сочетание:
- bullish structure;
- support hold;
- non-collapsing OI;
- supportive futures flow.

### `trend_down`
Зеркально наоборот.

### `range`
Используется, если:
- structure не даёт направленного преимущества;
- OI не расширяется;
- price oscillates внутри диапазона;
- breakout acceptance не подтверждён.

### `squeeze_prone_up`
Нужно одновременно:
1. price expansion up;
2. OI expansion up;
3. futures flow dominance up;
4. минимум одно дополнительное подтверждение:
   - funding tilt/extreme;
   - short liquidation burst;
   - supportive book imbalance;
   - явный perp-driven move with weak spot confirmation.

### `squeeze_prone_down`
Зеркальная логика.

### Конфликт `range + breakout`
Такой случай по умолчанию не должен автоматически давать `ENTER_*`.
Он должен сначала существовать как `WATCHLIST` / `WAIT`, пока breakout не переведёт рынок в accept/continuation state.

## Уточнения редакции v3

### Regime classes v1
Нужно жёстко поддержать:
- `trend_up`
- `trend_down`
- `range`
- `squeeze_prone_up`
- `squeeze_prone_down`
- `invalid`

### Правило для squeeze_prone
Недостаточно только `price + OI + funding`.
Нужны:
- price expansion;
- OI expansion;
- futures flow dominance;
- плюс хотя бы одно подтверждение:
  - funding tilt/extreme;
  - liquidation burst;
  - supportive book imbalance;
  - spot lag / perp-led imbalance.
