## Контекст
Этот файл является частью поэтапной реализации scanner-проекта на Bybit V5.

Работа по этому файлу должна вестись только после:
1. прочтения `0.md`;
2. прочтения последнего отчёта по предыдущему roadmap-файлу;
3. закрытия всех хвостов из предыдущего отчёта.

Если хвосты предыдущего шага не закрыты, реализацию этого файла начинать нельзя.

---

## Что должно быть получено по итогам подэтапа
- raw liquidations
- USD normalization
- 1m/5m buckets

## Технические требования
- Код должен оставаться модульным и читабельным.
- Не допускается смешивать в одном большом файле unrelated logic из разных слоёв.
- Все timestamps хранить и передавать в UTC.
- Все решения, влияющие на scanner state и decision logic, должны быть объяснимыми и тестируемыми.
- Везде, где это применимо, нужно учитывать, что архитектура уже сейчас проектируется под BTCUSDT, ETHUSDT и SOLUSDT, хотя активным остаётся только BTCUSDT.

## Требования к рыночным данным
- WebSocket-first: не добавлять лишние REST-запросы без явной причины.
- Spot и linear обрабатывать строго раздельно.
- Для minute OI использовать локальное накопление после запуска сервиса.
- Любая derived-логика должна быть объяснима, тестируема и пригодна для reason object.

## Дополнительные правила по liquidation feed

- Отсутствие новых liquidation events не должно автоматически считаться ошибкой, так как поток может естественно молчать.
- Нужно различать «событий не было» и «feed stale».
- Liquidation layer может участвовать как подтверждение squeeze/trap, но не должен всегда быть hard requirement для всех setup-типов.

## Уточнения редакции v3

### Обязательная нормализация liquidation events
Для каждого liquidation event нужно приводить:
- сторону;
- цену;
- размер;
- USD notional.

### Bucket outputs
На каждом bucket обязательны:
- `shortLiqUsd`
- `longLiqUsd`
- `totalLiqUsd`
- `liqImbalance`

Нужны отдельные 1m и 5m buckets.


## Уточнения редакции v5

### Data quality and anomaly handling
- входящие payloads должны проходить schema validation;
- битые/неполные сообщения не должны попадать в derived state;
- duplicate events должны быть отфильтрованы до записи в БД;
- невозможные значения (negative size, impossible price jump, broken timestamps) должны маркироваться как anomalies;
- anomaly не должна автоматически рушить процесс, но обязана логироваться и влиять на health/degraded markers.

### Latency tracking
- для каждого сообщения нужно хранить event time, receive time и processing time;
- должны считаться ingestion lag и processing lag;
- lag должен быть доступен для health layer и UI/debug panels.
